# 进展打卡反馈

本文件用于用户主动报告"刚吃了 X / 刚练完 Y / 刚测了体重"等阶段性进展时，LLM 输出**4 段固定结构反馈**（估算 / 评估 / 风险 / 补救）。

## 触发关键词

按用户输入识别意图分类：

| 类型 | 关键词样例 |
|---|---|
| **餐后** | 吃了 / 吃完 / 喝了 / 早餐 / 午餐 / 加餐 / 晚餐 / 刚吃 / 喝完 |
| **训练后** | 练完 / 做完 / 跑了 / 跳了 / 心率 / 燃了 / 训练完 / 拉伸完 |
| **测量** | 测了 / 体重 / 腰围 / 体脂率 / 称完 |

混合情况（如"刚吃完午饭训练完一身汗"）按"先训后餐"处理：先评估训练 → 再评估餐。

## 餐后 check-in 流程

### 1. 解析与估算

- 解析用户描述的食物（数量 / 烹饪方式 / 配料）
- 估算 kcal + 蛋白 / 脂肪 / 碳水（参考 `lifestyle-tips.md` 区块 A/B 已有数据；不在表里的食物按 LLM 常识估算 + 加"约"）
- 如果同会话内本日已有其他餐记录，**累加到当日总和**

### 2. 评估（三档）

| 档位 | 触发条件 |
|---|---|
| 🟢 **良好** | 单餐 kcal 在该餐预算 ±15% / 蛋白质达到该餐目标 / 无禁忌违反 |
| 🟡 **可优化** | 单餐 kcal 超 +15-30% 或不足 -15-30% / 蛋白质不足 / 碳水脂肪比例略失衡 / 无紧急风险 |
| 🔴 **需调整** | 单餐 kcal > +30% / 触发 profile 禁忌（乳糖不耐却吃奶 / 不吃香菜却点了菜含香菜 / 等）/ 含糖油混合物（油条 / 手抓饼 / 烤冷面 / 蛋糕）/ 当日累计已超 TDEE |

### 3. 风险检查（按顺序，触发即列）

- **profile 禁忌违反**：检查 `diet_pattern.restrictions`，列具体冲突项
- **当日 < 1200 kcal 趋势**：剩余 kcal < 600 而剩余餐次 < 2 时警示（参考 `kcal-targets.md` 1200 底线）
- **蛋白质不足**：当日累计 < 0.8 g/kg 趋势
- **糖油混合物预警**：用户报告吃了油条 / 烧饼 / 油炸糖食（参考修贤"避开糖油混合物"）
- **luteal 期高钠**：黄体期吃了过多咸的（火锅 / 麻辣烫汤底全喝 / 重口味卤味）→ 提醒水肿风险
- **menstrual 期生冷**：经期吃了冰饮 / 刺身 / 凉拌生菜 → 提醒可能加重不适
- **ED 信号**：用户说"今天就吃这一顿""什么都不想吃""吃完特别罪恶"——立即触发 `cycle-rules.md §跨阶段共通原则` ED 警示，**停止 kcal 评估** + 共情 + 建议加餐 + 必要时建议心理咨询

### 4. 补救建议（具体到下一步）

- 下一餐怎么调（具体食物 + 调整方向）
- 当日剩余 kcal 怎么分配
- 是否需要补蛋白 / 加饮水 / 加散步抵消（**注意**：抵消式心态有风险，慎推；只在用户主动问时给）

### 5. 餐后输出模板（固定 4 段，280-400 字）

```
📊 估算
- 约 [X] kcal
- 蛋白 [P]g / 脂肪 [F]g / 碳水 [C]g
- 累计今日：[Y] / [今日目标] kcal（剩余约 [Z] kcal）

🟢/🟡/🔴 评估
[一句话判断，含 phase / status 适配]

⚠️ 风险（无则跳过本节，不要写"无风险"）
- [具体风险 + 简短原因]
- [...]

💡 后续建议
- 下一餐（[预计时间]）：[具体食物组合 + 大致 kcal]
- 今日剩余预算：[...]
- 一句重点：[当前 phase / status 的特别提示]
```

## 训练后 check-in 流程

### 1. 解析与估算

- 解析用户描述的训练（类型 / 时长 / 强度信号如心率）
- 按 `kcal-targets.md §4` 训练燃烧表估算 kcal（按 `weight_kg / 60` 缩放）
- 如果用户给了心率：> 80% 最大心率（最大≈ 220 - age）= 高强度

### 2. 评估（三档）

| 档位 | 触发条件 |
|---|---|
| 🟢 **完美** | 强度匹配当前 phase 推荐档位 / 时长在合理区间 / 无体感警示 |
| 🟡 **可优化** | 强度比 phase 推荐高 1 档（如 luteal 推中等但用户做了高）/ 时长超 / 用户提到轻微不适 |
| 🔴 **需调整** | 强度比 phase 推荐高 2 档以上 / menstrual D1-3 做了 HIIT / 命中受伤部位 / 严重体感警示（头晕、心慌、关节痛） |

### 3. 风险检查（按顺序）

- **phase 不匹配**：
  - HIIT in `menstrual` D1-3 → 红色警示 + 建议明日改散步
  - HIIT in `luteal` 末段（D24+）→ 黄色提醒 + 减半建议
  - 高强度 in `peri_menopause` 反复出现 → 加重潮热风险
- **受伤史命中**：用户报告"做了深蹲 + 膝盖响"且 `body_status.injury_history` 有膝盖记录 → 红色 + 立即停训 + 必要时医生
- **过训信号**：心率 > 90% max 持续 / 训练时长 > 60 min HIIT / 一周连续 5 天高强度无休
- **体感警示**：用户提到头晕 / 心慌 / 异常关节痛 / 极端疲惫 → 立即停 + 补水 + 必要时医生
- **空腹有氧违规**：用户报告"早上空腹跑了 60 min" → 提醒不超 30 min（参考 `weekly-split.md §空腹有氧使用规范`）

### 4. 补救建议（具体到下一步）

- **训练后 30 分钟内**：补水（推荐量参 `kcal-targets.md §3.6`）/ 蛋白质补充（约 25-35 g 蛋白餐 / 蛋白粉）/ 静态拉伸 5-8 分钟
- **今晚**：睡眠优先（高强度训练后睡 7-8 小时是主要恢复手段）/ 是否需要泡澡放松
- **明天训练**：是否调整强度 / 改类型（如今天 HIIT，明天改稳态有氧）/ 间隔 48 小时再练相同肌群

### 5. 训练后输出模板（固定 4 段，280-400 字）

```
📊 估算
- 训练消耗：约 [Z] kcal（[时长] [类型]）
- 强度档：[低 / 中 / 中高 / 高]
- [心率分析（如用户给了心率）]

🟢/🟡/🔴 评估
[一句话判断，含 phase / status 适配判断]

⚠️ 风险（无则跳过本节）
- [具体风险 + 简短原因]
- [...]

💡 后续建议
- 30 分钟内：[补水 X mL / 补蛋白 Y g / 拉伸动作 1-2 个]
- 今晚：[睡眠 / 泡澡 / 是否再吃点]
- 明天训练：[强度调整 / 改类型 if any]
```

## 测量数据 check-in（体重 / 腰围 / 体脂率）

### 评估原则（不算单次，看趋势）

- **不评价单次数字好坏** —— 体重周内波动 1-2 kg 多是水分（黄体期尤甚）
- 如果用户记录到 4 周以上，**对比 4 周前 vs 现在**给趋势判断
- 4 周内的数据：只确认记录 + 鼓励继续记，不下结论
- **绝对不说"恭喜瘦了 / 你怎么又长了"**——数字 ≠ 用户价值

### 输出模板（更短，150-200 字）

```
📊 已记录
- [指标]：[数值]（[变化 if 有 4 周以上数据]）

📈 趋势观察（仅 ≥ 4 周数据有）
[一句话趋势描述，看 4 周窗口]

💡 一句重点
- 体重周内 1-2 kg 波动多是水分，看 2-3 周趋势更靠谱
- [当前 phase 特别提示，如 luteal 易显高]
```

## 通用硬约束（三种 check-in 共用）

- **不开玩笑评价数字**："你这吃得有点多哦~""今天又破纪录了！"——这种轻佻语气**禁止**
- **不强行抵消**：不主动建议"吃多了去多跑 30 min 抵消"——这是 ED 思维，损害身心
- **不否定用户体感**：用户说"我今天感觉很疲劳"，**不质疑**（"才练 30 min 怎么会累？"），先共情再建议
- **保持评估透明**：估算 kcal 一律加"约"；评估档位用 emoji + 一句话，不展开心理学说教
- **总长度 280-400 字**（餐后 / 训练后），**150-200 字**（测量），不超过

## 与其他文件的协作

- `kcal-targets.md` — 估算 kcal / 营养素 / 消耗的公式来源
- `cycle-rules.md` / `non-cyclic-rules.md` — phase 适配判断
- `weekly-split.md` — 训练强度 vs phase 是否匹配的判据
- `lifestyle-tips.md` — 食物 kcal 估算参考
- `exercises.md` — 受伤史命中时的禁忌动作对照
- `influences.md` — 修贤"放纵不等于失败"理念用于偶尔超 kcal 时的安抚
- `state/profile-schema.md` — `diet_pattern.restrictions` / `body_status.*` 必读用作风险检查
